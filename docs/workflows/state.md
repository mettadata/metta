# State

Reference for metta's on-disk state model — every directory and file the framework reads or writes, with a schema pointer and lifecycle note for each.

## Design stance

metta has no database. Durable state is a flat filesystem tree under two roots:

- `.metta/` — per-project runtime state (config, execution state, locks, logs).
- `spec/` — authored specifications, in-flight changes, archives, issues, backlog, and gaps.

Git is the transaction log. Every artifact mutation, spec merge, and archive is intended to land as a conventional commit on a feature branch, with `/metta-ship` performing the final merge.

Every YAML file that metta reads or writes is validated by a strict Zod schema in `src/schemas/` before a read returns or a write lands. The `StateStore` class (`src/state/state-store.ts`) is the one chokepoint: `read(path, schema)` parses YAML then calls `schema.safeParse(...)`; `write(path, schema, data)` validates `data` before serializing. A `StateValidationError` is thrown on any mismatch. Raw markdown artifacts bypass the schema layer (`readRaw`/`writeRaw`) because they are free-form authored content.

## Directory tree

```
<project-root>/
  CLAUDE.md                     # AI orchestrator context; regenerated by /metta-refresh
  .claude/
    commands/                   # installed slash commands (one .md per skill)
    hooks/
      metta-guard-edit.mjs      # PreToolUse guard; blocks edits outside an active change
    statusline/
      statusline.mjs            # optional status line integration
    settings.json               # registers hooks + statusLine with Claude Code
  .metta/
    config.yaml                 # project config (git-tracked)
    state.yaml                  # per-user runtime state (git-ignored)
    local.yaml                  # local overrides (git-ignored)
    state.lock                  # advisory lock file (git-ignored)
    logs/                       # per-invocation logs (git-ignored)
    .gitignore                  # declares the four git-ignored entries above
  spec/
    project.md                  # the constitution
    changes/
      <change-slug>/            # one directory per active change
        .metta.yaml             # change metadata
        intent.md               # artifact files (names vary by workflow)
        ...
    specs/
      <capability>/
        spec.md                 # living capability specification
        spec.lock               # hashed snapshot + reconciliation state
    archive/
      <YYYY-MM-DD>-<slug>/      # finalized or abandoned changes
        ...                     # preserved artifact set + gates.yaml
    issues/
      <slug>.md                 # logged issues
      resolved/
        <slug>.md               # resolved-and-archived issues
    backlog/
      <slug>.md                 # prioritized backlog items
      done/
        <slug>.md               # completed backlog items
    gaps/
      <slug>.md                 # reconciliation gaps (spec vs code)
```

## Top-level files

### `spec/project.md` — the constitution

The source of truth for project identity, stack, conventions, constraints, and off-limits patterns. Read by every skill that needs context (propose, plan, execute, verify, ship, refresh).

- **Authored by**: `metta-discovery` during `/metta-init`.
- **Updated by**: `/metta-refresh` when regeneration is triggered (managed sections only).
- **Read by**: every skill; `/metta-check-constitution` validates active changes against it; `CLAUDE.md` is regenerated from it.
- **Format**: free-form markdown with well-known section headings — `Project`, `Stack`, `Conventions`, `Architectural Constraints`, `Quality Standards`, `Off-Limits`.
- **Deleted by**: never deleted by metta.

### `CLAUDE.md` — AI orchestrator context

Regenerated from `spec/project.md`, the installed skill inventory, and the active-specs table. The `Metta Workflow` section is managed — the regenerator preserves human-authored content outside it.

- **Authored by**: `/metta-refresh`.
- **Read by**: Claude Code (and other AI orchestrators) as session-level context.
- **Deleted by**: never deleted by metta.

### `.metta/config.yaml` — project config

The project-level metta configuration. **Git-tracked.** Schema: `ProjectConfigSchema` in `src/schemas/project-config.ts`. Fields (all optional, strict object):

| Field | Schema | Purpose |
|-------|--------|---------|
| `project` | `ProjectInfoSchema` — `name`, `description?`, `stack?`, `conventions?` | Project identity. `name` is required inside the object. |
| `defaults` | `{ workflow, mode }` with `workflow` defaulting to `standard` and `mode` to `supervised` | Fallbacks when a skill does not specify. |
| `providers` | `Record<string, ProviderConfigSchema>` — `{ provider, model?, api_key_env? }` | AI provider configuration. |
| `tools` | `string[]` | Tool allowlist. |
| `gates` | `Record<string, GateConfigSchema>` — `{ command, timeout?, required?, on_failure? }` | Overrides for gate definitions. `on_failure ∈ { retry_once, stop, continue_with_warning }`. |
| `git` | `GitConfigSchema` — `{ enabled, commit_convention, commit_template?, protected_branches, merge_strategy, snapshot_retention, create_pr, pr_base }` | Git behavior; defaults are conservative (ff-only, conventional commits, protected main/master). |
| `docs` | `DocsConfigSchema` — `{ output, generate_on, types }` | Doc generation; defaults to `./docs`, `finalize`, standard doc types. |
| `auto` | `AutoConfigSchema` — `{ max_cycles, ship_on_success }` | `/metta-auto` loop behavior. |
| `context_sections` | `string[]` | Custom `CLAUDE.md` sections. |
| `adapters` | `string[]` | Installed AI-tool adapters. |
| `cleanup` | `{ log_retention_days }` | Log rotation. |

The bootstrap `metta install` writes a minimal version with only `project.name`, `project.description`, and `project.stack` populated (empty strings for the latter two).

- **Authored by**: `metta install`.
- **Updated by**: `/metta-init` after discovery; manual edits.
- **Read by**: `ConfigLoader` (`src/config/config-loader.ts`), every command that needs project defaults.
- **Deleted by**: never.

### `.metta/state.yaml` — per-user runtime state

Transient state scoped to the current workstation. **Git-ignored** (declared in `.metta/.gitignore`). Schema: `StateFileSchema` in `src/schemas/state-file.ts`.

```
{
  schema_version: number,
  execution?: ExecutionStateSchema,
  auto?: AutoStateSchema,
}
```

- `execution` — when `/metta-execute` is running, records the active `change`, `started` timestamp, `batches` (ordered `ExecutionBatch` entries each holding `tasks` with per-task status, commit SHA, worktree path, gate map, and `deviations` per the four-rule system), and top-level `deviations`. See `ExecutionStateSchema` in `src/schemas/execution-state.ts`.
- `auto` — when `/metta-auto` is driving the full loop, records `description`, `started`, `max_cycles`, `current_cycle`, and per-cycle `AutoCycle` entries (phase, artifacts, `batches_run`, optional verification counts). See `AutoStateSchema` in `src/schemas/auto-state.ts`.

- **Authored by**: `/metta-execute`, `/metta-auto`.
- **Read by**: the same commands on resume; `/metta-status` and `/metta-progress` may surface fields for display.
- **Deleted by**: never automatically — commands overwrite or clear sections at transition points.

### `.metta/local.yaml` — local overrides

Per-machine overrides not suitable for the team config. **Git-ignored.** No schema enforced today; reserved.

- **Read by**: `ConfigLoader` where applicable.
- **Deleted by**: never automatically.

### `.metta/state.lock` — advisory lock

An advisory file lock used by `StateStore.acquireLock(lockFile, timeout = 5000)` (`src/state/state-store.ts`). The lock convention places a lock file at `<state-file>.lock` alongside the file being guarded (for example `changes/abc.yaml.lock`). Stale locks older than `STALE_LOCK_THRESHOLD_MS` (60 seconds) are reclaimed automatically. **Git-ignored.**

- **Authored by**: whichever command is writing guarded state.
- **Deleted by**: the releasing command via the returned `release` closure (best-effort `unlink`).

### `.metta/logs/` — invocation logs

Per-command diagnostic output. **Git-ignored.** Rotated according to `config.cleanup.log_retention_days` (default 30).

- **Authored by**: any command emitting structured logs.
- **Deleted by**: log rotation sweep; never committed.

### `.metta/.gitignore`

Declares the four always-ignored paths: `state.yaml`, `local.yaml`, `logs/`, `state.lock`. Written by `metta install` with the `wx` flag so an existing file is left alone.

## `spec/changes/<change-slug>/` — active work

One subdirectory per in-flight change. `ArtifactStore` (`src/artifacts/artifact-store.ts`) is the only module that creates, updates, archives, or abandons these directories.

**Slug derivation.** `ArtifactStore.createChange(description, workflow, artifactIds, baseVersions?)` slugifies `description` by lowercasing, replacing non-alphanumerics with `-`, stripping a small stop-word list (`a`, `an`, `the`, `add`, `and`, `or`, `for`, `to`, `of`, `with`, ...), and truncating to 30 characters.

### `.metta.yaml` — change metadata

Schema: `ChangeMetadataSchema` in `src/schemas/change-metadata.ts`. Strict object — additional fields reject.

| Field | Type | Purpose |
|-------|------|---------|
| `workflow` | `string` | Name of the workflow (`quick`, `standard`, `full`, or custom). |
| `created` | ISO-8601 datetime | Creation timestamp. |
| `status` | `ChangeStatusSchema` = `active \| paused \| complete \| abandoned` | Lifecycle flag. |
| `current_artifact` | `string` | ID of the currently in-progress or next-ready artifact. |
| `base_versions` | `Record<string, string>` | Per-capability `spec.lock.hash` snapshots taken at change start. Used to detect upstream spec drift at finalize time. |
| `artifacts` | `Record<string, ArtifactStatus>` | Status per artifact ID. `ArtifactStatusSchema` = `pending \| ready \| in_progress \| complete \| failed \| skipped`. |

On `createChange`, every artifact in the workflow is seeded `pending`; the first artifact is flipped to `ready`, and `current_artifact` is set to it. `ArtifactStore.markArtifact(change, id, status)` updates one entry and advances `current_artifact` when the new status is `ready`, `in_progress`, or `complete`.

### Artifact markdown files

One markdown file per workflow stage that has produced output so far. Names are workflow-dependent; common artifacts include `intent.md`, `stories.md`, `spec.md`, `design.md`, `tasks.md`, `implementation.md`, `review.md`, `verification.md`, `summary.md`. See [`artifacts.md`](artifacts.md) for the catalog of 11 artifact types, the agent responsible for each, and the required sections.

Artifacts are written via `ArtifactStore.writeArtifact(change, fileName, content)` → `StateStore.writeRaw(...)`. They are free-form markdown — no Zod validation, but the authoring subagent is expected to produce real content (placeholder strings like `"intent stub"` are forbidden per the `Metta Workflow` rule in `CLAUDE.md`).

- **Authored by**: the subagent bound to each workflow stage (see [`agents.md`](agents.md)).
- **Read by**: downstream stages, review gates, `/metta-status`, `/metta-progress`, and the finalizer when merging deltas.
- **Deleted by**: never inside `spec/changes/` — the entire change directory is moved to `spec/archive/` on finalize or abandon.

## `spec/specs/<capability>/` — living specs

The merged-and-versioned capability specifications.

### `spec.md` — the specification

Free-form markdown. Each capability lives in its own directory so sibling files (`spec.lock`) can accompany it. Updated on `/metta-ship` when `SpecMerger` (`src/finalize/spec-merger.ts`) folds delta specs authored inside a change into the living spec.

- **Authored by**: the spec author subagents during a change (`metta-spec-author`, etc.).
- **Merged by**: `SpecMerger.merge(changeName, baseVersions, dryRun)`.
- **Read by**: the context engine, `/metta-import`, `/metta-fix-gap`, and any skill that needs current behavior.
- **Deleted by**: never automatically.

### `spec.lock` — version + reconciliation snapshot

Schema: `SpecLockSchema` in `src/schemas/spec-lock.ts`.

| Field | Type | Purpose |
|-------|------|---------|
| `version` | positive int | Monotonically increasing version. Bumped by `SpecLockManager.update(...)`. |
| `hash` | string | Hash of the parsed spec (see `hashSpec` in `src/specs/spec-parser.ts`). Used as the `base_versions` snapshot in change metadata. |
| `updated` | ISO-8601 datetime | Last write time. |
| `status` | `draft \| reviewed \| approved` (optional) | Review state. |
| `source` | `scan \| manual \| change` (optional) | Where the lock content came from. Defaults to `change` in `createFromParsed`. |
| `scanned_from` | `string[]` (optional) | For `source: scan`, the source files scanned. |
| `uncovered_behaviors` | non-negative int (optional) | Count of unscoped behaviors detected during reconciliation. |
| `reconciliation` | `ReconciliationSchema` (optional) | `{ verified_at, requirements: [{ id, status, gaps?, evidence? }] }` where `status ∈ { verified, partial, missing, unimplemented, diverged, undocumented }`. |
| `requirements` | `SpecLockRequirementSchema[]` — `{ id, hash, scenarios: string[] }` | Per-requirement hashes so drift is detectable without diffing prose. |

`SpecLockManager` (`src/specs/spec-lock-manager.ts`) owns read/write/update; `getBaseVersion(capability)` returns the hash string (or `null` when the lock is missing) and is called by `/metta-propose` to seed `change.base_versions`.

- **Authored by**: `SpecLockManager.update(capability, parsedSpec)` — called during spec merge and by `/metta-import` (`source: scan`).
- **Read by**: `/metta-propose` (for base versions), `/metta-fix-gap`, `/metta-verify`, the reconciler.
- **Deleted by**: never automatically.

## `spec/archive/<YYYY-MM-DD>-<slug>/` — finalized and abandoned changes

`ArtifactStore.archive(name)` moves `spec/changes/<name>/` to `spec/archive/<YYYY-MM-DD>-<name>/` via an `fs.rename`. `ArtifactStore.abandon(name)` updates `status: abandoned` first, then moves to `spec/archive/<YYYY-MM-DD>-<name>-abandoned/`. The original artifact set is preserved verbatim.

The finalizer (`src/finalize/finalizer.ts`) additionally writes `archive/<dir>/gates.yaml` containing:

```yaml
finalized_at: <ISO-8601>
all_passed: <bool>
results:
  - gate: <name>
    status: <pass|fail|warn|skip>
    duration_ms: <number>
```

`GateResultSchema` in `src/schemas/gate-result.ts` defines the per-result shape (the finalizer writes a reduced projection).

- **Authored by**: `/metta-ship` (via `Finalizer.finalize`) or `/metta-abandon`.
- **Read by**: `/metta-progress` history view; rarely read by skills.
- **Deleted by**: never automatically.

## `spec/issues/` — logged issues

Plain markdown files. `IssuesStore` (`src/issues/issues-store.ts`) owns reads and writes. Each issue file starts with `# <title>` and a bold-labelled frontmatter block:

```markdown
# <title>

**Captured**: <YYYY-MM-DD>
**Context**: <optional>
**Status**: logged
**Severity**: critical | major | minor

<free-form description>
```

Slugs are lowercased with non-alphanumerics replaced and truncated to 60 characters.

- **Authored by**: `/metta-issue`.
- **Read by**: `/metta-fix-issues`, `/metta-progress`.
- **Moved by**: `/metta-fix-issues --remove-issue` relocates the file into `spec/issues/resolved/<slug>.md` once the resolving change ships.
- **Deleted by**: never automatically.

### `spec/issues/resolved/<slug>.md`

Same format; records the resolution. Kept as a durable record of historical issues.

## `spec/backlog/` — prioritized items

Plain markdown files. `BacklogStore` (`src/backlog/backlog-store.ts`) owns reads and writes. Each item:

```markdown
# <title>

**Added**: <YYYY-MM-DD>
**Source**: <optional>
**Status**: backlog
**Priority**: high | medium | low

<free-form description>
```

- **Authored by**: `/metta-backlog add`.
- **Read by**: `/metta-backlog list`, `/metta-progress`, `/metta-next`.
- **Moved by**: `metta backlog done <slug>` relocates to `spec/backlog/done/<slug>.md`.
- **Deleted by**: never automatically.

## `spec/gaps/` — reconciliation gaps

Plain markdown files for tracked spec-vs-code divergence. `GapsStore` (`src/gaps/gaps-store.ts`) owns reads and writes. Each gap file uses `## <section>` headings:

```markdown
# Gap: <title>

## Status
claimed-not-built | partial | built-not-documented | diverged

## Source
## Claim
## Evidence
## Impact
## Related Spec
## Action
```

- **Authored by**: `/metta-import` (via the reconciler) and `/metta-check-constitution` when drift is detected.
- **Read by**: `/metta-fix-gap`, `/metta-progress`.
- **Deleted by**: `/metta-fix-gap` removes the file when the resolving change ships.

## The PreToolUse guard hook

`.claude/hooks/metta-guard-edit.mjs` (template: `src/templates/hooks/metta-guard-edit.mjs`) is installed by `metta install` and registered in `.claude/settings.json` under `hooks.PreToolUse` with matcher `Edit|Write|NotebookEdit|MultiEdit`. It blocks those four tool calls when no active metta change exists.

**Algorithm.** The hook reads the tool invocation from stdin, ignores non-guarded tools, and shells out to `metta status --json`. The status payload has two shapes: `{ change: "..." }` when a change is active, or `{ changes: [], message: "..." }` when none is. If either `status.change` is a string or `status.changes` is a non-empty array, the hook exits 0 and the edit proceeds. Otherwise it emits a nudge to stderr and exits 2, which Claude Code surfaces to the user:

```
metta-guard: <Tool> blocked — no active metta change.
Start one with /metta:quick <description> or metta quick <description>.
Then retry the edit.
Emergency bypass: disable this hook in .claude/settings.local.json.
```

Tolerance is deliberate: any error running `metta status` (CLI missing, not a metta project, timeout) is swallowed with `exit 0`, so the hook never blocks bootstrap or non-metta repositories. The intended emergency bypass is the per-user `settings.local.json` override — the hook does not support an inline flag.

The installer is idempotent: it detects an already-registered entry by searching `hooks.PreToolUse` for any command string containing `metta-guard-edit.mjs` and skips re-registration.
