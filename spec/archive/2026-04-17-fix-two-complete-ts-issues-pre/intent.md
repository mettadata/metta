# fix-two-complete-ts-issues-pre

## Problem

Two open issues share a root cause: `metta complete` accepts bad artifacts and punts validation to finalize-time, long after downstream artifacts have been written against incomplete data.

**Issue 1 — stub-placeholder artifacts accepted (`metta-complete-accepts-stub-placeholder-artifacts-on-intent-`)**: `metta complete intent` / `metta complete verification` check only that the artifact file exists and is non-empty. An orchestrator (or a confused author) can write a file containing the literal string `"intent stub"` and the command happily marks it complete. This was triggered in-session earlier and motivated logging the issue.

**Issue 2 — stories-valid runs only at finalize (`stories-valid-gate-catches-missing-us-fields-only-at-finaliz`)**: when `metta complete stories` runs, there is no validation that the stories document actually parses and has required fields on each US entry. Downstream stages (spec, research, design, tasks) get authored against malformed stories and the problem only surfaces at `metta finalize` — dozens of commits too late.

Both fixes belong in the same file (`src/cli/commands/complete.ts`), so batch them.

## Proposal

Add a pre-complete validation pass inside `complete.ts`'s action handler, between the file-exists check and the `markArtifact` call.

**Content sanity check (applies to all markdown artifacts generated by text stages — NOT wildcard generators like `**/*`):**
- Read the artifact file.
- Reject if its stripped length is < 200 bytes.
- Reject if it contains any of: `"intent stub"`, `"summary stub"`, `"spec stub"`, `"research stub"`, `"design stub"`, `"tasks stub"`, `"stories stub"`, `"architecture stub"`, `"verify stub"`, `"domain-research stub"`, `"ux-spec stub"` (case-insensitive).
- Reject if its H1 heading contains the literal placeholder `{change_name}` (means the template was committed without filling in).
- Rejection emits exit code 4 with a clear error naming the detected problem.

**Stories-valid pre-complete gate:**
- When `artifactId === 'stories'`, after the content sanity check, invoke the same validation path that `stories-valid` uses at finalize time.
- If validation fails, emit the same structured failures that finalize would surface, with exit code 4.
- Pass → proceed as usual.

## Impact

- `src/cli/commands/complete.ts` — add a new pre-complete validation block. ~30 lines added.
- `tests/` — new tests covering: stub content rejection, too-short content rejection, unfilled-template-heading rejection, stories-valid pre-complete rejection on bad stories, happy-path passthrough on good content.
- No schema changes. No new CLI surface. Existing callers that pass good content are unaffected.

## Out of Scope

- Validation of non-markdown artifacts (e.g. the implementation stage's wildcard generates).
- Retroactive validation of already-completed changes in `spec/archive/`.
- Restructuring the stories-valid logic itself — only invoking it earlier.
- Any changes to `metta finalize` (the finalize-time gate still runs, defense in depth).
