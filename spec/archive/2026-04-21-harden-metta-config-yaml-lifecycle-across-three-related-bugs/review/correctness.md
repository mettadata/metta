# Correctness Review: harden-metta-config-yaml-lifecycle-across-three-related-bugs

**Verdict**: PASS_WITH_WARNINGS

## Summary

Implementation satisfies the overwhelming majority of the spec's MUSTs: the schema lands first (Task 1.1), `setProjectField` is idempotent and comment/flow-preserving, `loadYamlFile` hard-fails via `ConfigParseError`, the doctor `--fix` path reads the raw file bytes and bypasses `ConfigLoader`, install writes stacks via the shared helper, and the preAction hook correctly gates every CLI command except the documented exemptions (install/init/doctor/update/completion). Two warnings are worth calling out: (1) the verifier-agent first-run heuristic diverges from spec R7's literal wording (checks `spec/archive/` too, and compares directory-emptiness rather than inspecting for `stories.md`/`intent.md`); (2) the `ConfigParseError` field is named `path` rather than the spec-mandated `filePath`. Neither is a blocker — the persona guidance is stricter than spec (fail-safer), and `path` matches the tests plus is callable/validated.

## Findings

### Critical

None — every MUST/MUST NOT traces back to code that implements it.

### Warnings

- `src/templates/agents/metta-verifier.md:25-29` + `.claude/agents/metta-verifier.md:25-29` — Spec R7 defines the first-run heuristic as "no user-authored change … determined by inspecting `spec/changes/` for directories containing a `stories.md` or `intent.md` file" (`spec.md:233-235`). The persona instead uses "BOTH `spec/changes/` and `spec/archive/` are empty". Two divergences: (a) the persona also gates on `spec/archive/`, which spec.md never mentions for this heuristic; (b) the persona checks directory emptiness, not the presence of `stories.md`/`intent.md` files. Consequence: a project that has shipped changes (archive non-empty) but currently has no active changes will be treated as "legacy → hard error" by the persona, while spec.md scenario 2 (`spec.md:255-261`) would mark it as first-run and default to `tests_only`. Design.md explicitly describes the stricter "both dirs empty" rule (`design.md:172-173`), so the implementation follows design, but design itself diverged from spec. Recommend updating either the persona to match spec (prefer), or updating the spec scenario to match the persona's stricter rule.

- `src/config/config-loader.ts:8-16` — Spec requires `ConfigParseError` to carry a `filePath: string` field (`spec.md:67-69`). The implementation uses `path: string` instead. `tests/config-loader.test.ts:153` and `src/cli/helpers.ts:149-155` both read `.path`, so the code is internally consistent, but the literal spec field name is not honored. Rename to `filePath` (or expose it as an alias getter) to match spec wording without breaking the existing `handleError` integration.

- `src/config/repair-config.ts:39-48` — Spec R4(b) calls for deduplication of "all duplicate map keys **anywhere in the document**" (`spec.md:104-106`); design.md `§Component 3` says "walks all `YAMLMap.items` in reverse". The implementation only dedups the top-level map and the first level of nested maps. The review prompt confirms the intended depth is "one level deep", but a strict reading of spec R4(b) is broader. For the known schema this gap is mostly academic (`providers.<name>.<field>` would be the only practical 2-level path), yet duplicate keys inside e.g. `providers.anthropic` would survive a repair pass. Either (a) extend the walker to recurse unconditionally, or (b) tighten the spec wording to match design ("one level deep"). Flagging because the review guidance asked us to check walk depth explicitly.

- `src/cli/commands/install.ts:178-180` — On a config that already has duplicate `project:` keys from the legacy regex writer, `setProjectField` now throws (via the `doc.errors.length > 0` guard in `config-writer.ts:16-18`) before the new `stacks:` can be written. The error message is raw ("Failed to parse ..."), with no mention of `metta doctor --fix`. Install is correctly exempt from the preAction hook (`src/cli/index.ts:96-102`), but it can still land in this failure mode on its own. The spec does cover it (`spec.md:42-43` explicitly defers pre-existing corruption to `metta doctor --fix`), but the user-facing message could be improved to direct the user to the repair command. Non-blocking because the scenario is narrow, but worth a follow-up.

- `src/cli/commands/instructions.ts:75-83` — The verification context injection calls `ctx.configLoader.load()` a second time (the preAction hook already loaded it via a fresh `ConfigLoader` instance in `src/cli/index.ts:113-115`). Because each instance caches independently, this re-reads and re-parses the YAML file. Not a correctness issue — both reads will succeed because the preAction hook only lets the command proceed when parsing succeeds — but it is a minor double I/O. Optional cleanup: share the preAction loader through the CLI context, or memoize.

The SKILL.md parity test exists (`tests/skill-structure-metta-init.test.ts:40-45`), but there's no automated guard for `metta-verifier.md` drift. Follow-up (not a spec-level MUST).
- Test coverage for the three batches is solid: idempotency (`tests/cli.test.ts:102-117`), doctor --fix dedup+auto-commit (`tests/cli.test.ts:459-521`), corrupt-config boundary (`tests/cli.test.ts:523-567`), instructions verification context (`tests/cli.test.ts:1642-1680`), config-loader throwing (`tests/config-loader.test.ts:141-155`).
