# Research: adaptive-workflow-tier-selection-emit-complexity-score-after

## Decision: Single-module `src/complexity/` (scorer, parser, renderer) invoked at three prompt sites in `complete.ts` with YAML-driven workflow diff for up/down-scaling, shared `askYesNo` helper in `helpers.ts`, and `--auto` flag persisted as `auto_accept_recommendation` in change metadata.

### Approaches Considered

#### File-Count Parsing (from research-file-parsing.md)

1. **remark unified AST walk** (selected) — Parse intent.md and summary.md via `unified().use(remarkParse)`, locate section headings by walking `tree.children`, collect `inlineCode` nodes whose values match a file-path heuristic (extension-anchored: `.ts`, `.yaml`, `.md`, etc., or prefix-anchored: `src/`, `tests/`, `dist/`). Deduplicate and count. Zero new dependencies; matches the existing pattern in `spec-parser.ts` and `stories-parser.ts`. Handles all three real-world Impact section formats (structured bullets, prose with inline refs, bare prose) with graceful-zero degradation when no file paths are found. Robust section-boundary detection because the AST handles heading-depth and trailing-whitespace variance.
2. **Regex heuristic on raw markdown string** — Extract the section by splitting on `\n## ` then apply a capture regex. Fewer lines of code (~20 vs ~60) but breaks the project's established markdown-parsing idiom; brittle against heading depth variance (e.g., `### Impact`) and trailing whitespace; produces a silent wrong slice on misidentified boundaries. Inconsistent with all other markdown consumers in the codebase.
3. **Structured convention (bullet format requirement)** — Require the Impact section to list files as `- \`path/to/file.ext\` — description` bullets and parse only list-item leading inline-code nodes. Near-perfect accuracy for compliant Format A intents, but systematically returns zero for the 30-40% of intents that use prose descriptions, and hard-fails for the current active intent (which uses bold module labels). Imposes an authoring constraint that no gate currently enforces.

#### Auto-Downscale (from research-auto-downscale.md)

1. **Insert block inside `complete.ts` gated on `artifactId === 'intent'`, after `markArtifact`, before next-artifact routing** (selected) — All required state (workflow name, score, artifact list, TTY flag, `--json` flag) is already in scope. Matches the existing `artifactId === 'stories'` and `artifactId === 'spec'` per-artifact gate pattern at lines 88 and 113. Single-commit boundary: both the `markArtifact` write and the `updateChange` downscale write happen before the existing auto-commit block at line 198, so the commit message `docs(<changeName>): complete intent` covers both atomically.
2. **Shared post-intent lifecycle hook** — Requires a new hook registry, hook wiring into the command loop, and explicit threading of the TTY/json prompt channel into the hook API. Only one consumer exists; the abstraction overhead is disproportionate.
3. **Separate `metta downscale` command** — Breaks atomicity (intent completes in one command, downscale fires in a second), creates a window where state is "intent complete, workflow not adjusted," and requires skills to chain two commands for one user-visible action.

For the **workflow collapse mechanism**: re-load the target workflow YAML via `workflowEngine.loadWorkflow('quick', searchPaths)` (already cached in the `WorkflowEngine.workflows` Map after the first call), derive `newArtifacts` by mapping `quickGraph.buildOrder` entries and carrying forward statuses from the existing map, then call `updateChange({ workflow: 'quick', artifacts: newArtifacts })`. This keeps the YAML files as the single source of truth and avoids hardcoding the planning artifact list. The alternative (surgically deleting keys by a hardcoded list of planning artifact names) produces a collapse that silently drifts as `standard.yaml` evolves.

For **prompt I/O**: native `node:readline` `createInterface` with a shared `askYesNo(question: string, defaultYes?: boolean): Promise<boolean>` helper extracted into `src/cli/helpers.ts`. The existing `install.ts` already contains this exact pattern (lines 92-100 of that file). Suppress when `!process.stdin.isTTY || program.opts().json`. No new dependencies. Third-party prompt libraries (inquirer, prompts) are ESM-compatible but unjustified for a single yes/no interaction.

#### A. Intent-time Auto-Upscale (symmetric to downscale)

1. **Same `complete.ts` insertion point, same YAML-diff mechanism, load target workflow and insert missing stages as `pending`** (selected) — Fully symmetric to the downscale approach. After `markArtifact('intent', 'complete')`, if the complexity score recommends a higher tier than the current workflow, fire a `[y/N]` prompt (default No — upgrading is a larger commitment). On Yes: load the target workflow graph, compute the diff (artifact IDs present in target but absent from current artifacts map), insert them as `pending`, call `updateChange({ workflow: targetName, artifacts: mergedArtifacts })`. Preserve all existing artifact statuses. For v1, only upscale to `standard`; gate `full` promotion behind an existence check of its stage templates, because the `full-workflow-references-missing-template-files-domain-resea` issue documents that `full.yaml` references templates that do not yet exist. This is a hard runtime guard: before allowing upscale to `full`, verify the relevant template files are present via `ctx.artifactStore.artifactExists` or a direct `existsSync` check on the builtin templates path.
2. **Pre-compute upscale targets at propose time and embed them in `.metta.yaml`** — Store the "would upscale to" workflow name at creation time so `complete.ts` does not need to re-derive it. Adds complexity to the propose/quick commands for a benefit that is already achieved by the scorer running at intent-complete time. The scorer must run at intent-complete time regardless (score is based on the content of intent.md which does not exist at propose time), so pre-computation is moot.

#### B. Post-Implementation Upscale + Retro-generation

1. **Mark `stories` and `spec` artifacts as `pending` in the new workflow, print a directive instructing the orchestrator to run `metta instructions stories` and `metta instructions spec` retroactively; exit 0** (selected) — The metta CLI runs in a Node.js process and cannot directly spawn Claude subagents. The `metta-product` and `metta-specifier` personas exist only as orchestrator-layer skills, not as in-process callables. The correct boundary is: the CLI's responsibility ends at state mutation and clear instruction output; the orchestrator (skill) picks up from there. Concretely: after the user confirms upscale at implementation-complete time, the CLI re-loads the target workflow (same YAML-diff mechanism as upscale A), inserts `stories` and `spec` as `pending` with correct dependency ordering from the graph's `buildOrder`, calls `updateChange`, then prints to stdout: `"Post-impl upscale accepted. Run: metta instructions stories --change <name>  then  metta instructions spec --change <name>. Verification resumes after both are complete."` The orchestrating skill reads this directive and invokes the appropriate subagents. This maintains the CLI/orchestrator boundary established by the rest of the codebase and avoids any IPC or subprocess coupling.
2. **Pre-author `stories.md` and `spec.md` in-process using templated skeletons populated from `intent.md` and `summary.md` content, mark both complete** — Executes in a single command without orchestrator involvement, but produces low-quality artifacts: the stories and spec would be structural shells without real product thinking or requirements engineering. Given the `metta-complete-accepts-stub-placeholder-artifacts-on-intent-` issue (which this project actively guards against with `MIN_CONTENT_BYTES` and `STUB_MARKERS` checks), auto-generated skeletal artifacts that pass these guards only by volume would violate the project's quality intent. Marking them `complete` without agent review undermines the verification gate. Not recommended.

The **trigger point** for concern B is inside `complete.ts` gated on `artifactId === 'implementation'`, after `markArtifact`, before the next-artifact routing block — mirroring the pattern used at the intent trigger point. Recompute the file count from `summary.md` `## Files` section using the same remark AST parser as concern E. If the recomputed tier exceeds the current workflow tier, fire the `[y/N]` prompt.

#### C. `--auto` / `--accept-recommended` Flag Propagation

1. **Add `--auto` boolean option to `propose`, `quick`, and `fix-issue` via Commander; persist as `auto_accept_recommendation: boolean` in `ChangeMetadataSchema`; read at each prompt site in `complete.ts`** (selected) — Commander `.option('--auto', 'Auto-accept tier recommendations')` with `.alias('--accept-recommended')` is the standard pattern. Both `propose.ts` and `quick.ts` pass the flag through to `createChange`; `fix-issue.ts` does the same if it creates a change. `createChange` accepts a new optional parameter `autoAccept?: boolean` that writes the field to `.metta.yaml`. At each prompt site, guard: `if (metadata.auto_accept_recommendation) { print one-line banner to stderr; return true }` before any `askYesNo` call. The schema extension requires relaxing `.strict()` on `ChangeMetadataSchema` to `.strict()` with the new optional field added — `ChangeMetadataSchema` currently uses `.strict()` (line 30 of `change-metadata.ts`), so the field must be explicitly declared there or `.strict()` must be changed to `.passthrough()`. Declaring it explicitly is the correct approach to maintain type safety. The `workflow_locked` field recommended by `research-auto-downscale.md` (section 4c) should be added in the same schema extension.
2. **Pass `--auto` as a per-invocation CLI flag on `metta complete` instead of persisting in state** — This would require the orchestrating skill to track and re-pass the flag on every `complete` invocation throughout the change lifecycle, which is error-prone. Persisting in state once at change creation and reading it at each prompt site is simpler and more reliable.

#### D. Banner Rendering + Code Sharing with `metta status`

1. **New module `src/complexity/renderer.ts` with two pure functions: `renderBanner(score): string` and `renderStatusLine(score): string`, plus `isScorePresent(metadata): boolean`** (selected) — Pure functions with no I/O dependencies; easily unit-tested. `renderBanner` produces a single-line advisory string (e.g., `[COMPLEXITY] 7 files — recommended: standard`); `renderStatusLine` produces multi-line output suitable for `metta status` human mode. Both return empty string when `score` is null/undefined, centralizing the null-guard in one place. `metta instructions` writes `renderBanner(score)` to `process.stderr` so `--json` stdout stays machine-readable (the existing banner-to-stderr pattern is already established in `instructions.ts` line 70 and `complete.ts` line 159). `metta status` uses `renderStatusLine` in human mode and passes the raw score object through in JSON mode. The `color()` helper from `helpers.ts` is used inside `renderer.ts` for ANSI formatting, consistent with the existing `agentBanner` and `banner` functions.
2. **Inline the banner string into each call site** — Duplicates the null-guard logic and ANSI formatting in at least three places (`instructions.ts`, `status.ts`, `complete.ts`). Any future change to banner format requires edits across all consumers. Not recommended.

#### E. Score Computation Module Shape

1. **`src/complexity/scorer.ts` with `scoreFromIntentImpact(intentMd): ComplexityScore | null` and `scoreFromSummaryFiles(summaryMd): ComplexityScore | null`, both delegating to shared `src/complexity/file-count-parser.ts`** (selected) — Both functions share the same remark AST walk logic (differ only in which heading they search for: `## Impact` vs `## Files`). Isolating the walk in `file-count-parser.ts` means the heading-label is the only parameter that varies between the two call sites. The `tierFromFileCount(n): 'trivial'|'quick'|'standard'|'full'` function captures the tier thresholds in one location. `ComplexityScore` is a plain interface: `{ score: number; signals: { file_count: number }; recommended_workflow: Tier }`. Pure functions throughout; no class state; no I/O. Imported by `complete.ts` at the three prompt sites (intent-complete, implementation-complete, and score-emission-to-instructions). Tested in isolation using markdown string fixtures, matching the existing test pattern in `src/specs/spec-parser.test.ts`.
2. **Single monolithic scorer function with inline parsing** — Collapses the walk logic into each scorer function, making them harder to test independently and impossible to reuse the walk for future signals (e.g., counting test files separately from source files). The `file-count-parser.ts` extraction costs ~10 lines of indirection and saves significant duplication if signals expand.

The module hierarchy within `src/complexity/`:
- `file-count-parser.ts` — remark AST walk, section-heading parameter, extension/prefix discriminator
- `scorer.ts` — `scoreFromIntentImpact`, `scoreFromSummaryFiles`, `tierFromFileCount`, `ComplexityScore` interface
- `renderer.ts` — `renderBanner`, `renderStatusLine`, `isScorePresent`

All three are pure, no-I/O modules. `complete.ts` and `instructions.ts` import from `scorer.ts` and `renderer.ts`. `status.ts` imports from `renderer.ts` only.

### Rationale

The combined architecture holds together on four axes:

**Cohesion.** All complexity logic lives in `src/complexity/`. No complexity reasoning leaks into `complete.ts`, `instructions.ts`, or `status.ts` beyond a function import and a call. Future signals (story count, test coverage ratio, dependency graph depth) can be added to `scorer.ts` and `file-count-parser.ts` without touching the CLI command layer.

**Reuse.** The single `file-count-parser.ts` remark walk is shared across `scoreFromIntentImpact` and `scoreFromSummaryFiles`. The single `askYesNo` helper in `helpers.ts` is shared across the three prompt sites in `complete.ts` and the existing prompt in `install.ts`. The single `renderer.ts` is shared across `instructions.ts` and `status.ts`. The single YAML-diff mechanism (load target workflow, map `buildOrder`, carry forward statuses) is reused across downscale, upscale at intent, and upscale at implementation — three sites, one algorithm.

**Fit with existing patterns.** Markdown parsing via `unified().use(remarkParse)` is the project idiom (`spec-parser.ts`, `stories-parser.ts`). Schema extension via Zod is the project idiom for all state reads and writes — `ChangeMetadataSchema` receives two new optional fields (`complexity_score`, `auto_accept_recommendation`) with explicit declarations, not `.passthrough()`. CLI flags via Commander `.option()` is the project idiom for all commands. Prompts via `node:readline` are already present in `install.ts`. Banners to `process.stderr` in JSON-compatible mode are already the pattern in `complete.ts` and `instructions.ts`. No new runtime dependencies are introduced.

**Boundary preservation for concern B.** The post-implementation retro-generation approach (print directive, exit 0, let the orchestrator skill spawn subagents) respects the established CLI/orchestrator division of responsibility. The CLI is an imperative shell that mutates state and emits instructions; the skill layer is where AI subagent orchestration happens. Collapsing this boundary by pre-authoring artifacts in-process would produce lower-quality artifacts and undermine the stub-content guard that already exists in `complete.ts`.
