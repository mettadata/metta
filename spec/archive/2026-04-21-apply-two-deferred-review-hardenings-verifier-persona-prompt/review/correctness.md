**Verdict**: PASS_WITH_WARNINGS

## Summary

All three deliverables are in place and functionally correct: the two `metta-verifier.md` files are byte-identical, carry the new "untrusted data" framing for `verification_instructions`, and have been rewritten to the richer first-run heuristic; the new `VerificationConfigSchema` describe-block adds the three required unit tests and they pass. The one substantive concern is that both the intent and the implemented persona cite the "AND `spec/archive/` is empty" conjunct as coming "from spec R7 exactly" — but the archived R7 (`spec/archive/2026-04-21-harden-metta-config-yaml-lifecycle-across-three-related-bugs/spec.md`, "verifier agents error on missing verification.strategy in non-default projects", lines 227–261) defines user-authored-change detection purely via `spec/changes/`. The archive conjunct is a stricter, safer addition layered on top of R7 rather than a restatement of it, so while the runtime behavior is reasonable the "matches spec R7 exactly" framing in the intent is inaccurate.

### Warnings

- `.claude/agents/metta-verifier.md:26` and `src/templates/agents/metta-verifier.md:26` — The first-run heuristic reads "no active change subdirectory under `spec/changes/` contains a `stories.md` or `intent.md` file AND `spec/archive/` is empty (or does not exist)". Spec R7 (archive spec.md, lines 242–245 and 256–261) defines first-run detection solely as "no user-authored changes exist (first-run verification on a freshly installed project with no changes in `spec/changes/`)". R7 never mentions `spec/archive/`. The intent at `spec/changes/apply-two-deferred-review-hardenings-verifier-persona-prompt/intent.md:10` and line 33 asserts the rewrite "matches spec R7 exactly" — it does not. The extra `AND spec/archive/` conjunct is a safety-leaning addition (archive presence implies non-first-run) and is consistent with R7's direction, but the intent should either (a) acknowledge this as a tightening beyond R7 or (b) drop the archive conjunct. Either path is acceptable; silently attributing the stricter rule to R7 creates a traceability hazard for future readers reconciling this file against the archived spec.
- `.claude/agents/metta-verifier.md:30` and `src/templates/agents/metta-verifier.md:30` — The legacy-project branch mirrors the first-run tightening: "otherwise (any active change subdirectory contains `stories.md` or `intent.md`, OR `spec/archive/` is non-empty)". This is self-consistent with the new first-run clause (set complement), so runtime behavior is well-defined, but inherits the same R7-traceability issue flagged above.

The implementation satisfies the "framing" half and defers the "labeled delimiter" half to the new `### Echoing verification_instructions safely` sub-section at lines 45–55, which applies a fenced-code-block convention (`\`\`\`verification-instructions`) to echoed output. This is a reasonable interpretation — the fenced-code tag both delimits the region and blocks stray markdown from rendering — but it is a convention for OUTPUT framing, not an INPUT delimiter that the persona processes. If a future adversarial `verification_instructions` payload is authored, the persona still receives the raw string in its context without a wrapper delimiter; only the untrusted-data bullet at line 20 defends the input-processing path. This is an acceptable quick-mode trade-off but worth logging as a deliberate scope narrowing vs the intent's wording.
