# fix-issue-orchestrator-asks-user-questions-are-answerable

## Problem

The metta orchestrator routinely pauses a change workflow to ask the user
questions whose answers are available in public documentation — for example,
"does Claude Code skill frontmatter support `context: fork`?" is a
deterministic question answered by `https://code.claude.com/docs/en/skills`,
yet the orchestrator asked the user instead of fetching the docs. Users then
have to explicitly redirect with "search the doco online and work it out."

The root cause is an instruction-surface gap, not a tool gap: `WebFetch` and
`WebSearch` are already available, but the only place the workflow prescribes
their use is a narrow "technology-option grounding during Round 1 discovery"
slot in `/metta-propose` and `/metta-quick`. There is no general rule — in
`CLAUDE.md` or the skill definitions — telling the orchestrator to resolve
research-phase questions from public docs before asking the user. Absent such
a rule, the default is `AskUserQuestion` because every skill's discovery
instructions push user interaction.

Affected: every AI-driven metta session. Every round-trip the orchestrator
spends asking the user about documented framework/API behavior is a wasted
interaction the user has to correct.

Reference: `spec/issues/orchestrator-asks-user-questions-that-are-answerable-by.md`.

## Proposal

Add a "Research Discipline" rule to the authoritative workflow primer so it
flows into `CLAUDE.md` via the existing `<!-- metta:workflow-start -->` /
`<!-- metta:workflow-end -->` marker section that `metta refresh` regenerates.

Edit `src/delivery/workflow-primer.ts` — specifically the `workflowPrimerLong()`
array — appending a new `### Research discipline` subsection after the existing
`### Forbidden` subsection with the following rule content:

- **(a) project-level scope:** the rule lives in the shared primer, so it
  appears in every `CLAUDE.md` regenerated by `metta refresh` and every
  scaffold written by `metta init` (the short variant inherits the same
  mandate, but the detailed rule lives in the long variant to keep the short
  primer short).
- **(b) research / design scope:** explicitly applies to research-phase and
  design-phase questions about external framework/API/tool documented
  behavior (examples: "does Claude Code support X feature", "what fields does
  the Y endpoint accept", "is the Z flag deprecated").
- **(c) escalation boundary:** only escalate to the user for subjective
  judgments — scope, cost, product direction, tradeoff choices — never for
  deterministic, documented facts.
- **(d) named tools:** cite `WebFetch` and `WebSearch` by name as the intended
  tools. `WebFetch` for known authoritative URLs, `WebSearch` to discover the
  authoritative source.

Then regenerate `CLAUDE.md` via `metta refresh` to propagate, and archive the
issue file with `git mv spec/issues/orchestrator-asks-user-questions-that-are-answerable-by.md spec/issues/resolved/`.

Update the `tests/delivery.test.ts` assertion set to pin the new rule into the
workflow primer output so a future edit that drops the rule fails the test
suite.

## Impact

- `src/delivery/workflow-primer.ts` — `workflowPrimerLong()` gains the new
  `### Research discipline` subsection. `workflowPrimerShort()` unchanged.
- `CLAUDE.md` — the auto-generated `## Metta Workflow` section grows by one
  subsection after `metta refresh` runs.
- `tests/delivery.test.ts` — one new `expect(formatted).toContain(...)` assertion
  and/or a new test that directly exercises `workflowPrimerLong()`.
- `spec/issues/orchestrator-asks-user-questions-that-are-answerable-by.md` —
  moves to `spec/issues/resolved/`.

No runtime behavior changes; this is a documentation-shaped change that
routes into orchestrator prompts.

## Out of Scope

- Editing individual skill `SKILL.md` templates (Candidate Solution 2 from the
  issue). The primer approach is higher-leverage and single-source.
- Introducing a dedicated `metta-doc-researcher` subagent (Candidate Solution
  3 from the issue). Not needed when the primer rule is in place.
- Adding runtime enforcement (a hook that blocks `AskUserQuestion` when a
  WebFetch answer is possible). Orchestrator judgment remains the enforcement
  surface; the rule just makes the judgment explicit.
- Editing `spec/project.md`. The "Metta Workflow" section in `CLAUDE.md` is
  generated from `workflow-primer.ts`, not from `spec/project.md`'s
  Conventions / Off-Limits, so project.md is not the authoritative source for
  this content.
