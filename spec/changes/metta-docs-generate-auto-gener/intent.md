# metta-docs-generate-auto-gener

## Problem

Metta accumulates authoritative project knowledge in two places: `spec/specs/` (9 living capability specs) and `spec/archive/` (8+ archived changes, each containing intent, spec, design, tasks, summary, and gates). This knowledge exists but is locked inside framework-internal YAML and Markdown files that are not human-navigable without understanding metta's directory conventions.

Developers working in a metta-managed project currently have no auto-generated, human-readable documentation that reflects the current state of the system. After a change is finalized, the specs are updated and the archive entry is written — but no corresponding docs output is produced. This means:

- New contributors must read raw spec files in `spec/specs/` to understand how the system behaves.
- Changelog information (what changed, why, when) is buried in `spec/archive/<change>/summary.md` files that accumulate with no aggregated view.
- Architectural decisions and system design from `spec/archive/<change>/design.md` are not surfaced to readers who are not already navigating the spec tree.
- API surface documented in specs is not compiled into an accessible reference.

The gap is that metta finalizes changes into specs but never materializes those specs into the output docs that humans and external tools actually read.

## Proposal

Implement `metta docs generate` as a template-based, deterministic doc generator that reads from `spec/specs/`, `spec/archive/`, and `spec/project.md` and writes four Markdown documents to the configured output directory (default `./docs`):

- **`architecture.md`** — compiled from `design.md` files in archived changes, grouped by capability, with ADR-style rationale preserved
- **`api.md`** — compiled from `spec.md` files in `spec/specs/`, covering requirements, Given/When/Then scenarios, and behavioral contracts for each capability
- **`changelog.md`** — compiled from `summary.md` files in `spec/archive/`, sorted by date descending, grouped by capability with what changed and why
- **`getting-started.md`** — compiled from `spec/project.md` (stack, conventions, constraints) plus an overview of capabilities drawn from living specs

Generation is template-driven: each doc type has a Handlebars-style template file (stored in `dist/templates/docs/`) that defines structure and section ordering. The generator fills templates from parsed spec data — no AI provider call, no network access, fully deterministic output.

Every generated file includes a metta-managed header block:

```markdown
<!-- Generated by Metta — do not edit directly -->
<!-- Source: spec/specs/auth/, spec/archive/2026-04-06-add-mfa/ -->
<!-- Run `metta docs generate` to regenerate -->
```

Automatic generation triggers on `metta finalize` when `generate_on: finalize` is set in config (the default). Manual generation is always available via `metta docs generate` and `metta docs generate <type>`. A `--dry-run` flag prints a diff of what would change without writing files.

Config lives in `.metta/config.yaml` under the `docs` key:

```yaml
docs:
  output: ./docs
  generate_on: finalize       # finalize | verify | manual
  types: [architecture, api, changelog, getting-started]
```

The `output` path is configurable to accommodate projects that already have a `docs/` structure and need to target a subdirectory.

## Impact

**Who benefits**: Any developer or team using metta on a project where the docs directory is committed and shared. The primary audience for generated docs is contributors who want to understand the system without reading raw spec internals.

**What changes operationally**:
- `metta finalize` gains a doc generation step that runs after spec merge and archive write, before the change is closed. No net new user action required.
- `docs/` directory is written (and committed, if git integration is enabled) after each finalize.
- `metta docs generate` becomes a first-class CLI command with subcommand routing per type.
- Templates for all four doc types are added to `dist/templates/docs/` as distributable framework assets.

**Downstream surfaces that benefit**:
- Generated `CLAUDE.md` `<!-- metta:reference-start -->` section already links to `docs/architecture.md`, `docs/api.md`, and `docs/changelog.md`. These links are currently dead for projects that have not manually written those files. After this change, the links resolve.
- `metta refresh` regenerates docs as part of its full refresh pass (since docs are a derived output).

**Blast radius**: Limited. This is additive — no existing spec, state, or workflow logic is modified. The `finalize` step gains a new trailing phase, but if doc generation fails it MUST NOT block the finalize from completing; a warning is emitted instead.

## Out of Scope

- **AI-assisted generation**: No AI provider calls. All content comes from parsing existing spec Markdown and filling templates. Summarization, inference, or content elaboration via LLM are out of scope for v1.
- **HTML or non-Markdown output formats**: Only `.md` files are generated. PDF, HTML site generation, or OpenAPI JSON are out of scope.
- **Watch mode / file-system triggers**: Docs do not auto-regenerate when spec files are edited directly. Regeneration is triggered by metta lifecycle events or explicit CLI invocation only.
- **Versioned doc snapshots**: Docs are always current-state. Generating per-release or per-tag snapshots is out of scope.
- **Custom doc types beyond the four defined**: The `types` config key accepts only `architecture`, `api`, `changelog`, and `getting-started`. Plugin-defined doc types are out of scope for v1.
- **Existing `docs/` file preservation**: Files in `docs/` that are not one of the four managed types are untouched. However, if a user manually edits a generated file (one that carries the metta-managed header), those edits will be overwritten on next generation. The header warns of this explicitly.
- **Doc generation during `metta ship`**: Ship merges the worktree branch into main. Doc generation fires at finalize, not ship. Generating docs twice (finalize then ship) is out of scope.
- **Interactive doc review or approval workflow**: Generated docs are written directly without a review gate. Spec approval gates are already enforced upstream in the finalize workflow.
