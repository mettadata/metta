# metta issue, metta backlog add, and other CLI state-mutating commands write to whatever branch is currently checked out, with no warning and no main-branch targeting. Observed 2026-04-18 during a zeus fix session: I ran `metta issue` three times and `metta backlog add` once from /home/utx0/Code/metta. Unbeknownst to me, HEAD was on a sibling-agent feature branch `metta/fix-issue-metta-refresh-leaves-claude-md-uncommitted-metta-r` (created earlier by /metta-fix-issues running on the refresh-uncommitted bug). All four entries silently landed on that feature branch instead of main. Impact: issue-tracker and backlog are conceptually main-branch artifacts that inform planning across all changes — contaminating a feature branch with them can cause lost entries on branch abandonment, merge conflicts, or entries getting tied to unrelated changes on squash. Expected behavior: (a) refuse to write with a warning if HEAD is not main, (b) offer to stash/checkout/restore automatically, (c) or write directly to main via a temporary worktree/stash-dance pattern. Silent contamination of another agent working branch is how parallel metta sessions step on each other. Affects: metta issue, metta backlog add/done, possibly metta gaps/specs writes. Not yet verified: whether metta propose/quick guard against being run while HEAD is on another feature branch.
