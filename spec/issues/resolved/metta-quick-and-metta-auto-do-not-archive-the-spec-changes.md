# /metta-quick and /metta-auto do not archive the spec/changes/<name>/ directory after shipping, leaving stale folders on main. Reproduced in this session: 6 features shipped via /metta-quick or /metta-auto (board-header, document-title-includes-card-count-prefix, export-filename-uses-active-board-name, focus-visible-outline, keyboard-shortcut-k-cycles-next-board, relative-time-badge-activityview) left their spec/changes/<name>/ dirs on main. By contrast, 10 features shipped via /metta-propose were correctly archived to spec/archive/2026-04-18-<name>/. Symptom: subsequent 'metta finalize' on a new change fails the stories-valid gate because it sees the accumulated stale folders. Driver has to manually clean up before shipping. Likely cause: the /metta-quick SKILL (or the 'metta quick'/finalize CLI path it runs) skips the archive-changes step that /metta-propose's finalize performs. Fix: ensure /metta-quick and /metta-auto terminal steps mv spec/changes/<name>/ to spec/archive/YYYY-MM-DD-<name>/ (matching /metta-propose behavior) before the git merge to main.
