# tasks.md completion-marking convention is undefined — metta-planner writes tasks.md with `### Task N.N:` headings and `**Done:**` criteria, but no checkbox field. metta-executor agents inconsistently retrofit `[x]` markers onto the heading (e.g. `### Task 1.1: ... [x]`) — some do, some do not. Observed 2026-04-18 during fix-metta-fix-issues-skill-contract change: Tasks 1.1, 1.2, 1.3, 1.4, 1.6 received `[x]` markers from their executors; Task 1.5 did not, because its executor correctly noted the file has no formal checkbox syntax. Net effect: tasks.md looks like Task 1.5 is unfinished when in fact its work shipped and was committed. The orchestrator (me, checking whether a task was done) had to cross-reference git log to confirm. Fix options: (1) standardize the tasks.md template with `- [ ]` checkbox bullets that executors flip to `- [x]`, (2) forbid executors from touching tasks.md at all and rely on `metta complete implementation` as the sole completion signal, (3) add `metta task done <task-id>` CLI so the framework owns the marker. Option 2 is cleanest — executors already mark `metta complete implementation`, which is the actual workflow advance; retrofitted `[x]` markers are decorative and error-prone. Severity: minor — cosmetic inconsistency, not a correctness bug, but causes confusion when an orchestrator or human reviews tasks.md mid-batch.
